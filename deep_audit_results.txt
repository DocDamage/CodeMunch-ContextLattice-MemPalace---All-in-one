# Technical Debt Audit and v1.0 Remediation Strategy

## Document Purpose

This document is the actionable technical-debt companion to the current implementation and release-planning docs.
Its job is not to restate every audit script result line-for-line.
Its job is to translate the audit signal into a coherent remediation order for the path to `v1.0`.

This report should be read alongside:
- `docs/implementation/REMAINING_WORK.md`
- `docs/implementation/LLMWorkflow_Post_0.9.6_Strategic_Execution_Plan.md`
- `docs/implementation/PROGRESS.md`
- `docs/releases/CHANGELOG.md`

---

## Executive Summary

The repository is no longer in danger of failing because it lacks features.
It is in danger of slowing down, regressing, or becoming harder to trust because too much of the implementation surface still carries prototype-era debt.

The audit signal is clear:
- the codebase is broad and capable, but structurally uneven
- operational trust is being weakened by silent-failure patterns and inconsistent module contracts
- several core modules have grown large enough to resist safe change
- newer ingestion work is moving in the right direction, but it is entering the same repo-wide discipline gap unless governance catches up

The correct move is not random cleanup.
The correct move is a staged hardening program with explicit release gates.

The highest-risk debt clusters are:
1. error suppression and weak failure visibility
2. monolithic files and helper duplication in core modules
3. missing PowerShell contract hygiene such as `Set-StrictMode`, `[CmdletBinding()]`, `.SYNOPSIS`, and `[OutputType()]`
4. portability and security debt such as hardcoded paths, `Invoke-Expression`, and heuristic secret findings
5. uneven test coverage and evidence quality around foundational modules

There are also meaningful positives in the current audit:
- no giant functions over the configured threshold were flagged
- no `$global:` state usage was flagged
- no alias abuse using `%` or `?` was flagged
- new engine-facing asset ingestion work is already landing with tests, which means the platform is still moving forward while hardening work begins

## Current Head Reality

The audit needs to be read in the context of current head, not an earlier snapshot of the project.
Recent work has already started to strengthen the mixed-artifact and game-asset side of the platform.

### Recently Landed or In Flight

The repository now includes concrete foundation work for broader game and engine asset support, including:
- engine-aware asset manifest categories and game preset scaffolding
- classification for spritesheets, tilemaps, plugins, RPG Maker assets, Unreal assets, Epic or Fab assets, and shared asset bundles
- Unreal descriptor extraction for `.uplugin` and `.uproject`
- RPG Maker asset catalog parsing for common asset families and plugin metadata
- regression tests covering the new asset-manifest, Unreal descriptor, and RPG Maker asset-catalog flows

That is good progress.
It also raises the quality bar.
The repo is no longer experimenting with whether mixed-artifact ingestion exists.
It is now responsible for making that ingestion maintainable, documented, and operationally trustworthy.

## How to Read This Audit

Not every finding class has the same meaning.
This report separates true release blockers from hygiene debt and from heuristic-only signals.

### High-Confidence Findings

These are direct, actionable issues that should be treated as real debt unless a specific case is intentionally justified:
- missing `Set-StrictMode`
- missing `[CmdletBinding()]`
- missing `.SYNOPSIS`
- missing `[OutputType()]`
- duplicate helper definitions
- empty `catch` blocks
- unapproved PowerShell verbs
- `Invoke-Expression` usage
- `Write-Host` usage inside reusable modules

### Heuristic Findings

These findings are valuable, but they require human review before being treated as defects:
- potential missing tests
- hardcoded secrets
- hardcoded absolute paths
- output suppression using `Out-Null` or `$null =`

### Scale Snapshot

The present audit evidence shows:
- `58` monolithic files over `1000` lines
- `32` files missing `Set-StrictMode`
- `19` duplicate function-name collisions
- `219` instances of `-ErrorAction SilentlyContinue`
- `4` empty catch-block locations
- `30` files with unapproved PowerShell verbs
- widespread help and contract gaps across top-level modules

This is not a one-sprint cleanup.
It is also not a reason to stop delivery.
It is a reason to sequence the work deliberately.

## Release Blockers by Priority

### Priority 0: Failure Visibility and Unsafe Execution

These are the most important issues to clear before `v1.0` certification.
They directly affect diagnosability, safety, and trust.

#### P0 Findings
- `219` uses of `-ErrorAction SilentlyContinue`
- `4` empty catch-block locations across `GoldenTasks.ps1`, `GeometryNodesParser.ps1`, `DoclingAdapter.ps1`, and `ExternalIngestion.ps1`
- `Invoke-Expression` usage in `audit4.ps1` and `Invoke-VulnerabilityScan.ps1`
- heavy `Write-Host` usage in modules that should prefer pipeline-safe output and structured diagnostics

#### Why This Matters
- silent failure breaks operator confidence and test signal
- swallowed exceptions make regressions look like success
- `Invoke-Expression` increases security and predictability risk
- `Write-Host` pushes the repo toward UI-style scripting instead of composable tooling

#### Exit Criteria
- every silent-failure site is either removed or explicitly justified with a comment and bounded fallback behavior
- empty catch blocks are eliminated
- `Invoke-Expression` is removed or isolated behind a documented, reviewed exception
- reusable modules prefer `Write-Verbose`, `Write-Information`, or object output

### Priority 1: Structural Refactoring of God Modules

The repo has too many large files for safe maintenance.
The problem is not just style.
The problem is reviewability, test isolation, ownership clarity, and regression radius.

#### Highest-Pressure Modules
- `MLModelDeploymentPipeline.ps1` at `2787` lines
- `ExternalIngestion.ps1` at `2732` lines
- `AIGenerationPipeline.ps1` at `2701` lines
- `GodotQuestExtractor.ps1` at `2613` lines
- `GoldenTasks.ps1` at `2545` lines
- `ExtractionPipeline.ps1` at `1405` lines
- `LLMWorkflow.GameFunctions.ps1` at `1197` lines

#### Why This Matters
- broad files accumulate hidden coupling
- helper reuse becomes copy-paste reuse instead of canonical reuse
- test fixtures become larger and slower to reason about
- new capabilities, including game-asset ingestion, risk inheriting the same structural debt pattern

#### Exit Criteria
- the largest modules are split into coherent private helper files or submodules
- duplicate helpers are replaced with canonical shared implementations
- each refactored surface has targeted tests at its new boundaries
- ownership of key modules becomes obvious from the file layout

### Priority 2: Module Contracts and PowerShell Hygiene

This is the largest broad-spectrum debt class.
It is less dramatic than silent failure, but it affects every developer and every downstream consumer.

#### Signal Highlights
- `32` files missing `Set-StrictMode`
- top offenders missing `[CmdletBinding()]` include `AIGenerationPipeline.ps1`, `entrypoint.ps1`, `DashboardViews.ps1`, and `NaturalLanguageConfig.ps1`
- top offenders missing `.SYNOPSIS` include `BlenderPythonParser.ps1`, `bootstrap-llm-workflow.ps1`, `LLMWorkflow.GameFunctions.ps1`, `UnrealDescriptorParser.ps1`, and `RPGMakerAssetCatalogParser.ps1`
- top offenders missing `[OutputType()]` include `AIGenerationPipeline.ps1`, `LLMWorkflow.GameFunctions.ps1`, `LLMWorkflow.psm1`, and `NaturalLanguageConfig.ps1`
- `30` files include unapproved PowerShell verbs

#### Why This Matters
- weak contracts make functions harder to discover and trust
- strict mode gaps let bugs survive longer than they should
- help gaps slow onboarding and reduce self-documentation quality
- inconsistent verbing makes the public surface feel improvised

#### Exit Criteria
- every reusable module starts with `Set-StrictMode -Version Latest` or a documented compatibility equivalent
- exported and reusable internal functions get `[CmdletBinding()]`
- public and semi-public functions get `.SYNOPSIS`
- exported functions get `[OutputType()]` where it improves predictability
- unapproved verbs are renamed or consciously grandfathered with documentation

### Priority 3: Portability, Security, and Release Discipline

These issues are partly heuristic, but they are still important for a trustworthy release.

#### Findings
- heuristic hardcoded-secret matches in `LLMWorkflow.HealFunctions.ps1` and `NaturalLanguageConfig.ps1`
- hardcoded absolute path signal in `entrypoint.ps1`, MCP deploy scripts, audit helpers, and scratch-style tooling
- potential test gaps around foundational modules like `AtomicWrite.ps1`, `CommandContract.ps1`, `FileLock.ps1`, `Journal.ps1`, `TypeConverters.ps1`, `Workspace.ps1`, and related runtime surfaces

#### Why This Matters
- `v1.0` should not depend on local-machine assumptions
- security reviews cannot dismiss secret-like strings without inspection
- release confidence depends on tests around state, locking, contracts, and workspace rules

#### Exit Criteria
- every heuristic secret finding is triaged and either removed, rotated, or documented as false positive
- hardcoded paths are migrated to `$PSScriptRoot`, config, or environment-bound resolution where appropriate
- core runtime primitives have direct tests or clear indirect coverage evidence

## Recommended Workstreams

### Workstream 1: Failure-Handling Sweep

Focus first on the places where the system can lie about its own health.

#### Immediate Targets
- `GoldenTasks.ps1`
- `GeometryNodesParser.ps1`
- `DoclingAdapter.ps1`
- `ExternalIngestion.ps1`
- modules with the heaviest `Write-Host` footprint such as `DashboardViews.ps1`, `LLMWorkflow.HealFunctions.ps1`, `MLModelDeploymentPipeline.ps1`, and `LLMWorkflow.Dashboard.ps1`

#### Deliverables
- no empty catches
- reduced silent-failure count
- structured logging guidance for module authors
- explicit exception policy for expected versus unexpected failures

### Workstream 2: Core Module Decomposition

This should target the highest-change, highest-risk files first rather than trying to split every large file at once.

#### First Candidates
- `MLModelDeploymentPipeline.ps1`
- `ExternalIngestion.ps1`
- `AIGenerationPipeline.ps1`
- `GoldenTasks.ps1`
- `ExtractionPipeline.ps1`
- `LLMWorkflow.GameFunctions.ps1`

#### Deliverables
- decomposed helper files with canonical ownership
- reduced duplicate helper footprint
- tests pinned to the new seams
- lower blast radius for future changes

### Workstream 3: Public Surface and Help Contracts

This is the governance track for module shape and discoverability.

#### First Candidates
- `LLMWorkflow.psm1`
- `LLMWorkflow.GameFunctions.ps1`
- `bootstrap-llm-workflow.ps1`
- `AIGenerationPipeline.ps1`
- `NaturalLanguageConfig.ps1`
- `DashboardViews.ps1`
- `UnrealDescriptorParser.ps1`
- `RPGMakerAssetCatalogParser.ps1`

#### Deliverables
- strict mode added where missing
- `[CmdletBinding()]`, `.SYNOPSIS`, and `[OutputType()]` applied to prioritized modules
- approved verb naming pass
- clearer help for newly added game-asset ingestion utilities so they do not become second-class surfaces

### Workstream 4: Test and Evidence Coverage Hardening

The repo now has valuable test momentum in newer asset-ingestion work.
That should become the norm, not the exception.

#### First Candidates
- `AtomicWrite.ps1`
- `CommandContract.ps1`
- `FileLock.ps1`
- `Journal.ps1`
- `TypeConverters.ps1`
- `Workspace.ps1`
- `StateFile.ps1`
- `LLMWorkflow.GameFunctions.ps1`

#### Deliverables
- direct tests for state, locking, and contract primitives
- explicit mapping between critical modules and test files
- fewer audit findings that rely on heuristic test inference

### Workstream 5: Security and Portability Review

This workstream converts heuristic warnings into verified outcomes.

#### First Candidates
- `LLMWorkflow.HealFunctions.ps1`
- `NaturalLanguageConfig.ps1`
- `entrypoint.ps1`
- deployment scripts for Blender and Godot MCP surfaces
- audit and scratch scripts that still encode local absolute paths

#### Deliverables
- secret false positives resolved or removed
- path resolution standardized
- `Invoke-Expression` usage eliminated or formally exempted
- clearer separation between production code, tooling, and scratch utilities

## Special Note on Mixed Artifact and Game Asset Ingestion

This area deserves explicit attention because it is both a strength and a risk multiplier.

The repo is actively expanding beyond text and code into engine descriptors, plugin metadata, spritesheets, tilemaps, and asset catalogs.
That is strategically correct.
It also means ingestion modules must not repeat the same historical debt pattern seen elsewhere in the repo.

### What This Means Practically
- new ingestion modules should ship with tests from day one
- parser modules should have help and command contracts early, not later
- descriptor inventory support and deep binary extraction support must remain clearly distinguished
- marketplace or engine-specific provenance rules should be designed before the surface grows further

### Immediate Governance Targets
- `UnrealDescriptorParser.ps1`
- `RPGMakerAssetCatalogParser.ps1`
- `SpriteSheetParser.ps1`
- `AtlasMetadataParser.ps1`
- `ExtractionPipeline.ps1`
- `LLMWorkflow.GameFunctions.ps1`

This is one of the places where the repo can improve while still growing.
We should keep that momentum, but with stronger discipline than earlier phases had.

## Strengths Worth Preserving

A good audit should not only point at flaws.
It should also identify where the project is already doing something right.

Current positive signals include:
- broad capability coverage across orchestration, extraction, packs, governance, and engine-facing tooling
- no flagged `$global:` state usage
- no flagged giant functions beyond the configured threshold
- no flagged alias abuse using `%` or `?`
- recent additions in the game-asset area landing with tests instead of relying entirely on later cleanup

These strengths matter because they show the repo is not in chaos.
It is in transition.
The hardening work is substantial, but it is tractable.

---

## v1.0 Readiness Gate

This audit recommends the following minimum bar before calling the repository `v1.0` ready:

1. no unresolved empty catch blocks in production modules
2. `Invoke-Expression` removed from production and security-sensitive paths
3. silent failure patterns materially reduced and justified where retained
4. top-tier monolithic modules decomposed enough to reduce review and testing risk
5. strict mode and function-contract hygiene applied to core public surfaces
6. foundational runtime modules have direct or clearly evidenced automated coverage
7. mixed-artifact and game-asset ingestion utilities are documented, tested, and clearly scoped
8. portability and secret heuristics are triaged to closure, not ignored

If those gates are not met, the repo may still be feature-rich, but it will not yet be release-trustworthy.
